## 1. المشكلة التقنية

### 1.1 استنزاف موارد السيرفر (Resource Exhaustion)

في أنظمة التجارة الإلكترونية، تُعد عملية الـ **Checkout** من أكثر العمليات استهلاكاً للموارد الحاسوبية، حيث تتضمن:

- **استعلامات قاعدة البيانات المتعددة**: قراءة بيانات السلة، المنتجات، المحفظة
- **عمليات الكتابة المتسلسلة**: إنشاء الطلب، تحديث المخزون، خصم الرصيد
- **الأقفال على مستوى الصفوف (Row-Level Locks)**: لضمان تكامل البيانات

عند وصول عدد كبير من الطلبات المتزامنة، يحدث ما يُعرف بـ **Resource Contention**، حيث تتنافس جميع الطلبات على نفس الموارد المحدودة:

```
┌─────────────────────────────────────────────────────────────────┐
│                    سيناريو الانهيار (Collapse Scenario)         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   [100 طلب متزامن]                                              │
│         ↓                                                        │
│   [Database Connection Pool: 10 اتصالات]                        │
│         ↓                                                        │
│   [90 طلب في الانتظار] ← تراكم في الذاكرة                       │
│         ↓                                                        │
│   [زيادة Latency لجميع الطلبات]                                 │
│         ↓                                                        │
│   [Timeout Errors + Memory Exhaustion]                          │
│         ↓                                                        │
│   [انهيار النظام بالكامل]                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 مشكلة Race Conditions

تحدث **Race Conditions** عندما تحاول عدة عمليات متزامنة الوصول إلى نفس المورد وتعديله في آن واحد. في سياق الـ Checkout:

| المشكلة | الوصف | النتيجة |
|---------|-------|---------|
| **Overselling** | بيع كمية أكبر من المخزون المتاح | خسارة مالية وسمعة |
| **Double Spending** | خصم الرصيد مرتين لنفس الطلب | شكاوى العملاء |
| **Lost Updates** | فقدان تحديثات بسبب الكتابة المتزامنة | بيانات غير متسقة |

### 1.3 تأثير الهجوم المتزامن (Concurrent Attack Impact)

عند إرسال 50 طلب متزامن بدون حماية:

```
الوقت (ثانية)    │ زمن الاستجابة (ms)  │ الحالة
─────────────────┼─────────────────────┼──────────────
0.0              │ 150                 │ طبيعي
0.5              │ 800                 │ بطء ملحوظ
1.0              │ 2500                │ بطء شديد
1.5              │ 5000                │ Timeout قريب
2.0              │ Connection Refused  │ انهيار
```

**المشكلة الجوهرية**: النظام يحاول معالجة جميع الطلبات بدون تمييز، مما يؤدي إلى استنزاف الموارد وتدهور الأداء للجميع.

## 2. الحل المعماري المطبق

### 2.1 فلسفة الحل: Slot-based Concurrency Control

اعتمدنا على نمط **Semaphore** المُوزَّع عبر **Distributed Database Locks** لتحقيق التحكم بالسعة:

```
┌─────────────────────────────────────────────────────────────────┐
│              Slot-based Concurrency Control                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   السعة القصوى: 3 Slots                                         │
│                                                                  │
│   ┌─────────┐  ┌─────────┐  ┌─────────┐                         │
│   │ Slot 1  │  │ Slot 2  │  │ Slot 3  │                         │
│   │ [مشغول] │  │ [مشغول] │  │ [متاح]  │                         │
│   └─────────┘  └─────────┘  └─────────┘                         │
│                                                                  │
│   [طلب جديد] ──► يأخذ Slot 3 ──► يُعالج ──► يُحرر Slot 3       │
│   [طلب زائد] ──► لا يوجد Slot ──► رفض فوري 429                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 لماذا اخترنا الرفض الفوري (Fail-Fast) بدلاً من الانتظار؟

| الاستراتيجية | السلوك | المزايا | العيوب |
|--------------|--------|---------|--------|
| **Queue & Wait** | الطلب ينتظر حتى يتوفر slot | لا يُفقد أي طلب | تراكم في الذاكرة، زيادة Latency |
| **Fail-Fast (429)** | الطلب يُرفض فوراً | حماية فورية للسيرفر | العميل يحتاج إعادة المحاولة |

**اخترنا Fail-Fast للأسباب التالية:**

1. **حماية قاعدة البيانات**: منع تراكم الاتصالات المفتوحة
2. **استقرار زمن الاستجابة**: الطلبات المقبولة تُعالج بسرعة طبيعية
3. **تجربة مستخدم أفضل**: الرفض الفوري أفضل من الانتظار الطويل
4. **Circuit Breaker Pattern**: حماية النظام من الانهيار التام

### 2.3 آلية Circuit Breaker

```
┌─────────────────────────────────────────────────────────────────┐
│                    Circuit Breaker Pattern                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   الحالة الطبيعية (Closed):                                     │
│   ─────────────────────────                                      │
│   [طلبات] ──► [معالجة] ──► [استجابة 200]                        │
│                                                                  │
│   عند الضغط العالي (Open):                                      │
│   ─────────────────────────                                      │
│   [طلبات زائدة] ──► [رفض فوري 429] ──► حماية النظام            │
│                                                                  │
│   الفائدة:                                                       │
│   - الطلبات المقبولة تعمل بكفاءة عالية                          │
│   - النظام لا ينهار تحت الضغط                                   │
│   - العملاء يعرفون فوراً أن عليهم إعادة المحاولة                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4 اختيار الحد الأقصى (MAX_CONCURRENT = 3)

تم اختيار الرقم 3 بناءً على:

| العامل | القيمة | التأثير |
|--------|--------|---------|
| **Database Connection Pool** | 10 اتصالات | كل checkout يستخدم ~2-3 اتصالات |
| **متوسط وقت المعالجة** | 500ms | 3 طلبات × 500ms = استخدام معقول |
| **هامش الأمان** | 30% | للعمليات الأخرى في النظام |

**المعادلة:**
```
الحد الأقصى = (Connection Pool Size / Connections per Request) × Safety Margin
           = (10 / 3) × 0.7
           ≈ 3 طلبات متزامنة
```

## 3. تفاصيل الكود البرمجي

### 3.1 البنية العامة للحل

```php
class OrderController extends Controller
{
    /**
     * الحد الأقصى للطلبات المتزامنة
     */
    private const MAX_CONCURRENT_CHECKOUTS = 3;

    /**
     * مفتاح الـ Cache لتتبع الـ slots
     */
    private const CHECKOUT_SLOTS_KEY = 'checkout_api_concurrent_slots';

    /**
     * مدة صلاحية الـ slot بالثواني (لمنع Deadlock)
     */
    private const CHECKOUT_SLOT_EXPIRATION = 60;
}
```

### 3.2 دالة الـ Checkout مع Concurrency Control

```php
public function checkout()
{
    /*
     * =================================================================
     * المرحلة 1: محاولة الحصول على Slot (Acquire)
     * =================================================================
     * 
     * نتحقق من وجود slot متاح قبل البدء بأي معالجة.
     * هذا يضمن عدم استهلاك موارد لطلبات سيتم رفضها.
     */
    $slotId = $this->tryAcquireCheckoutSlot();

    if ($slotId === null) {
        /*
         * Fail-Fast: رفض فوري بـ HTTP 429
         * 
         * لماذا 429 وليس 503؟
         * - 429 (Too Many Requests): يخبر العميل أن الحد تم تجاوزه
         * - 503 (Service Unavailable): يعني أن الخدمة معطلة
         * 
         * 429 أدق لأن الخدمة تعمل، لكن السعة ممتلئة.
         */
        Log::warning('Checkout rejected - max concurrent requests reached', [
            'user_id' => auth()->id(),
            'max_concurrent' => self::MAX_CONCURRENT_CHECKOUTS,
            'current_slots' => $this->getCurrentCheckoutSlotCount(),
        ]);

        return response()->json([
            'message' => 'Too many checkout requests. Server is busy.',
            'error_code' => 'CHECKOUT_RESOURCE_LIMIT_EXCEEDED',
            'max_concurrent_checkouts' => self::MAX_CONCURRENT_CHECKOUTS,
            'retry_after_seconds' => 5,
        ], 429);
    }

    /*
     * =================================================================
     * المرحلة 2: المعالجة مع ضمان التحرير (try-finally)
     * =================================================================
     */
    try {
        // المنطق الأصلي للـ Checkout
        if ($this->shouldUsePressureMode()) {
            return $this->checkoutBlackFriday();
        }
        return $this->checkoutNormal();

    } finally {
        /*
         * =================================================================
         * المرحلة 3: تحرير الـ Slot (Release) - مضمون التنفيذ
         * =================================================================
         * 
         * كتلة finally تضمن تنفيذ التحرير في جميع الحالات:
         * - نجاح العملية (return)
         * - فشل العملية (exception)
         * - أي مسار تنفيذ آخر
         * 
         * هذا يمنع الـ Deadlock الذي قد يحدث إذا لم يُحرر الـ slot.
         */
        $this->releaseCheckoutSlot($slotId);

        Log::info('Checkout slot released', [
            'user_id' => auth()->id(),
            'slot_id' => $slotId,
        ]);
    }
}
```

### 3.3 آلية إدارة الـ Slots

#### 3.3.1 محاولة الحصول على Slot

```php
private function tryAcquireCheckoutSlot(): ?string
{
    $slots = $this->getActiveCheckoutSlots();

    /*
     * التحقق من السعة المتاحة
     * 
     * إذا عدد الـ slots المشغولة >= الحد الأقصى
     * نرفض الطلب فوراً بإرجاع null
     */
    if (count($slots) >= self::MAX_CONCURRENT_CHECKOUTS) {
        return null;
    }

    /*
     * إنشاء slot جديد بمعرف فريد
     * 
     * uniqid() يضمن عدم التكرار حتى في الطلبات المتزامنة
     * نضيف timestamp للصلاحية (لمنع Deadlock)
     */
    $slotId = uniqid('checkout_slot_', true);
    $expiration = time() + self::CHECKOUT_SLOT_EXPIRATION;

    $slots[$slotId] = $expiration;

    /*
     * حفظ في الـ Cache (Distributed Lock)
     * 
     * نستخدم Cache بدلاً من متغير محلي لأن:
     * - يعمل عبر عدة Workers/Processes
     * - يبقى حتى لو أُعيد تشغيل العملية
     */
    Cache::put(self::CHECKOUT_SLOTS_KEY, $slots, self::CHECKOUT_SLOT_EXPIRATION * 2);

    return $slotId;
}
```

#### 3.3.2 تحرير الـ Slot

```php
private function releaseCheckoutSlot(string $slotId): void
{
    $slots = Cache::get(self::CHECKOUT_SLOTS_KEY, []);
    
    /*
     * حذف الـ slot من القائمة
     * 
     * unset() آمن حتى لو الـ slot غير موجود
     * (قد يكون انتهت صلاحيته تلقائياً)
     */
    unset($slots[$slotId]);

    Cache::put(self::CHECKOUT_SLOTS_KEY, $slots, self::CHECKOUT_SLOT_EXPIRATION * 2);
}
```

#### 3.3.3 تنظيف الـ Slots المنتهية (Deadlock Prevention)

```php
private function getActiveCheckoutSlots(): array
{
    $slots = Cache::get(self::CHECKOUT_SLOTS_KEY, []);
    $now = time();

    /*
     * تنظيف الـ Slots المنتهية الصلاحية
     * 
     * هذا يمنع الـ Deadlock في حالة:
     * - انهيار العملية قبل تحرير الـ slot
     * - خطأ في الشبكة منع التحرير
     * - أي سبب آخر لعدم التحرير
     * 
     * بعد CHECKOUT_SLOT_EXPIRATION ثانية، الـ slot يُحرر تلقائياً
     */
    $activeSlots = array_filter($slots, function ($expiration) use ($now) {
        return $expiration > $now;
    });

    /*
     * تحديث الـ Cache إذا تم حذف slots منتهية
     */
    if (count($activeSlots) !== count($slots)) {
        Cache::put(self::CHECKOUT_SLOTS_KEY, $activeSlots, self::CHECKOUT_SLOT_EXPIRATION * 2);
    }

    return $activeSlots;
}
```

### 3.4 أهمية كتلة try-finally

```
┌─────────────────────────────────────────────────────────────────┐
│                    try-finally Flow                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│   try {                                                          │
│       // سيناريو 1: نجاح                                        │
│       return $this->checkoutNormal();  ──────────┐              │
│                                                   │              │
│       // سيناريو 2: فشل                          │              │
│       throw new Exception("Error");  ────────────┤              │
│   }                                               │              │
│                                                   ▼              │
│   finally {                                                      │
│       // يُنفذ دائماً في كلا السيناريوهين                       │
│       $this->releaseCheckoutSlot($slotId);  ◄────────────────   │
│   }                                                              │
│                                                                  │
│   النتيجة: الـ Slot يُحرر دائماً = لا Deadlock                  │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**لماذا لا نستخدم try-catch فقط؟**

```php
// ❌ خطأ: قد لا يُحرر الـ slot
try {
    $result = $this->checkoutNormal();
    $this->releaseCheckoutSlot($slotId);  // لن يُنفذ إذا حصل exception
    return $result;
} catch (Exception $e) {
    $this->releaseCheckoutSlot($slotId);  // يُنفذ فقط عند exception
    throw $e;
}

// ✅ صحيح: finally يضمن التنفيذ دائماً
try {
    return $this->checkoutNormal();
} finally {
    $this->releaseCheckoutSlot($slotId);  // يُنفذ دائماً
}
```

## 5. الخلاصة

### 5.1 ملخص الحل

تم تطبيق **Slot-based Concurrency Control** على مستوى HTTP لحماية مسار الـ Checkout من الانهيار تحت الضغط العالي:

| المكون | الوظيفة |
|--------|---------|
| **MAX_CONCURRENT_CHECKOUTS** | تحديد السعة القصوى (3 طلبات) |
| **tryAcquireCheckoutSlot()** | محاولة الحصول على slot |
| **releaseCheckoutSlot()** | تحرير الـ slot بعد الانتهاء |
| **try-finally** | ضمان التحرير في جميع الحالات |
| **Slot Expiration** | منع الـ Deadlock |

### 5.2 الفوائد المحققة

1. **حماية السيرفر**: منع استنزاف الموارد
2. **استقرار الأداء**: زمن استجابة ثابت للطلبات المقبولة
